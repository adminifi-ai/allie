Allie / Consumer Contract Research + shape packet / 2026-06-19

Make Allie a portable evidence CLI with host-specific wrappers.

Consuming apps should not integrate with Allie as a GitHub-only app, a SaaS-first scanner, or a test-framework package. The durable contract is: one manifest in, one evidence packet out, deterministic exit semantics, and optional adapters for GitHub, Azure, local development, and later hosted review.

Recommendation

Best design: standalone Rust CLI plus manifest and reporter contract, distributed through a binary installer, npm shim, and container. GitHub Action and Azure YAML templates are thin wrappers around the same CLI.

Why it wins: it preserves local replay, host agnosticism, privacy controls, and stable evidence packets while still giving CI users a copy-paste path.

User Outcome

An application team can add Allie to any staged web app, run it locally or in CI, get actionable accessibility evidence, and publish reports without changing CI providers.

Primary Contract

allie.yml or .allie/manifest.yml defines target, startup, credentials profile, flows, discovery policy, standards profile, redaction, and reporters.

Outputs

Evidence JSON, HTML report, artifact directory, JUnit XML, SARIF, Markdown summary, CI-specific check payloads, and deterministic exit code.

Stop Rule

Do not build hosted dashboards or provider-specific actions before the same manifest can run locally, in GitHub, and in Azure with comparable receipts.

Alternatives

Couples Allie to language, package manager, and test runner. Makes Azure/local/non-JS adoption feel second-class. Support later as an SDK, not as the main contract.

Hosted SaaS scanner first Teams connect repo/preview URL and inspect a dashboard. Best long-term executive reporting and collaboration surface. Host-agnostic, locally replayable, privacy-friendly, aligned with evidence packet architecture. Requires crisp install story and strong examples, or it feels lower-level than an action. Recommended.

Manifest

All host-specific integrations should point at the same manifest. The manifest is the product surface consumers learn, review, diff, and tune.

target:
  base_url: http://127.0.0.1:4173
  start: npm run dev -- --host 127.0.0.1
policy:
  profile: wcag22-aa
  blocking_classes: [deterministic, scripted_required]
reporters: [html, json, junit, sarif, markdown]
flows:
  - id: homepage
    discover: true

CLI

The CLI owns execution, policy, browser worker orchestration, model policy, evidence writing, and exit code. Wrappers should not reinterpret the result.

allie verify \
  --config .allie/manifest.yml \
  --out .allie/runs/$ALLIE_RUN_ID

allie report \
  --packet .allie/runs/$ALLIE_RUN_ID/evidence.json \
  --format junit,sarif,markdown,github,azure

CI Wrappers

GitHub Action, Azure template, Docker image, and npm shim all reduce setup friction but remain adapters. They install Allie, call the CLI, publish artifacts, and surface summaries.

# GitHub: upload artifacts + optional SARIF
# Azure: PublishTestResults@2 + PublishPipelineArtifact@1
# Local: allie verify --config .allie/manifest.yml

Reporter Contract

Output Purpose Why it matters
evidence.json Canonical Allie packet. Stable replay, audit, release, and hosted-viewer boundary.
report.html plus artifacts Human drilldown for engineers, QA, and accessibility specialists. Works locally, in Azure artifacts, GitHub artifacts, or object storage.
junit.xml Generic CI test result channel. Azure supports JUnit/NUnit/VSTest/XUnit/CTest publication through PublishTestResults@2.
results.sarif Code scanning where host supports it. GitHub accepts third-party SARIF from Actions, external CI, or the API. Treat this as optional because not every accessibility issue maps cleanly to code locations.
summary.md Portable PR/build summary. Easy to paste into GitHub step summaries, Azure logs, Slack, Teams, or release notes.
github-check.json, azure-summary.json Host-specific projections. Generated from evidence, never the source of truth.

Rollout Plan

1. Contract hardening: write an ADR and schema for manifest, reporters, target identity, and exit codes.

2. Local first: implement allie init, allie verify, and reporter emission against Vanity.

3. CI parity: add GitHub Action and Azure YAML examples that call the same CLI and publish the same artifacts.

4. Serious dogfood: run Linejam as the first Next app integration, then Sploot once its current auth work lands.

5. Hosted viewer: only after packets are stable, add an optional server that ingests evidence packets.

Verification System

Claim: one consumer manifest can run locally, in GitHub, and in Azure-equivalent CI without changing Allie semantics.

Falsifier: a host wrapper needs different policy semantics, different findings, or a different canonical report.

Driver: run the same manifest against Vanity and Linejam through local shell, GitHub workflow, and Azure YAML sample.

Grader: schema-valid evidence packets, comparable findings, expected exit code, published HTML artifacts, JUnit result, optional SARIF upload.

Evidence packet: .allie/dogfood/<repo>/<mode>/ with HTML report, JSON packet, screenshots, trace, logs, JUnit, SARIF, and wrapper transcript.

Cadence: every wrapper change and before any hosted viewer work.

Research Notes

Source Relevant pattern Allie implication
axe-core and @axe-core/cli Core engine integrates into existing test environments; CLI emits JSON/stdout and can fail CI with exit code. Allie should expose deterministic CLI semantics and JSON first, not require a hosted integration.
Playwright accessibility docs Accessibility checks can be regular Playwright tests, with scan results attached to test output. Allie reporters should integrate with existing test artifacts, but keep Playwright behind the worker boundary.
pa11y-ci CI-centric CLI uses config files, URL lists/sitemaps, JSON reporter, custom reporters, and Docker examples. Manifest plus reporters is the right consumer primitive; container is a distribution option, not the product core.
Lighthouse CI CLI-centered CI suite with optional GitHub Action and server/reporting ecosystem. Follow the CLI-first ladder: local command, CI examples, then optional hosted evidence viewer.
GitHub SARIF upload GitHub can ingest third-party SARIF from Actions, external CI, or API. Allie should emit SARIF as an optional projection, not as the canonical packet.
Azure PublishTestResults@2 and Pipeline Artifacts Azure accepts generic test result formats and pipeline artifacts from any runner. Allie should emit JUnit plus artifact directories so Azure support is native without an Azure-only product branch.

Non-Goals For The Next Slice

Do not build a hosted dashboard, marketplace action polish, full SDK, auto-remediation PR bot, or CI provider abstraction layer before the core manifest and reporter contract are proven in at least one local static app and one real app with a dev server.
